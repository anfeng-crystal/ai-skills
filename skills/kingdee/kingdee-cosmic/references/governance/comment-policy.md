# 功能性注释规范

## 必须写注释的对象
- 新增业务类、插件类、工具类、枚举、跨模块复用组件。
- 新增公共方法、受保护方法、复杂私有方法。
- 涉及苍穹生命周期、事务、跨库保存、字段回写、BOTP、报表取数、工作流、异步任务的逻辑。
- 关键 BOS SDK 调用、DataSet 生命周期、权限/组织口径、保存目标、回写目标、失败补偿和幂等边界。

## 类注释
类注释至少说明：
- 这个类解决什么业务问题。
- 典型使用场景和挂载位置。
- 依赖的苍穹平台能力或元数据前提。
- 是否会写库、触发操作、跨对象回写或产生外部副作用。

## 方法注释
方法注释至少说明：
- 方法职责和调用时机。
- 参数含义，特别是字段标识、实体标识、事件参数来源。
- 返回值语义；无返回值时说明主要副作用。
- 异常或失败边界；事务、跨库、幂等、重复执行风险。
- 依赖平台事件顺序、SDK 签名或元数据字段时，说明稳定前提或证据来源，不把查证过程写进注释。

## 工具类说明
工具类必须说明：
- 适合在哪些业务对象或插件类型中使用。
- 不适合使用的场景。
- 关键方法的输入输出约定。
- 需要调用方自行保证的前置条件。

## 避免的注释
- 不写“设置变量”“调用方法”这类机械复述。
- 不写 AI 生成痕迹明显的空泛总结。
- 不把大段需求背景塞进代码；长说明放到需求文档或 reference。
- 不用注释替代类型、命名和方法拆分；如果代码没有任何注释会让读者看不出头绪，至少补类/方法/关键分支级说明。

## 注释与说明文档
- 注释和 skills 只写稳定信息，不写本次实施过程。
- 代码注释只写：职责、业务口径、平台约束、边界条件、长期有效但不直观的原因。
- skills、操作说明、示例说明只写：适用场景、前置条件、输入输出、固定步骤、失败条件、注意事项。
- 本次修改经过、排查路径、尝试失败、与用户沟通记录、验证经过、交付口径，只能写在交付说明、PR 描述或变更记录中。
- 出现“本次 / 此次 / 这里改为 / 为了兼容 / 先…再…最后… / 经过排查 / 最终采用”等过程性表述时，必须改写为职责、约束、边界或固定规则表述。
- 能通过命名、函数拆分、类型约束表达清楚的，不额外写注释。
