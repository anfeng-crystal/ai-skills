# Implement Feature

> **Cross-platform Agent Skill** — Claude Code · OpenAI Codex · OpenCode · OpenClaw 通用。
> 跨平台 SKILL.md，遵循开放 Agent Skill 规范。

## 触发边界
- 用户要求新增能力、补充业务逻辑、扩展接口、增加配置、改造流程或改变预期行为时使用。
- 只解释代码时改用 `explain-code`；只审查风险或给修改建议但不落地时改用 `review-code`；现有行为出错、回归或测试失败时改用 `fix-bug`。
- 混合请求中如果同时包含”先方案”和”直接实现”，先给最小方案和影响范围；用户明确要求直接改且范围局部时，可边实现边说明关键假设。

## 快速工作流
1. 明确目标：确认使用者、输入输出、状态变化、影响对象、兼容约束、验收方式和禁止改动范围。
2. 读近似实现并盘点现有能力：优先查同模块、同类型能力、现有扩展点、命名、错误处理、测试和配置方式；编码前先盘点仓库已有 helper、wrapper、模板、脚手架、SDK 封装、成熟三方库和标准服务，确认哪些可以直接复用。
3. 给实施方案：用 3-5 点说明落点、改动范围、关键数据流、失败边界、兼容影响和验证方式。
4. 方案门禁：默认等用户确认后编码；如果用户带着已审核方案明确要求实现，按方案执行，不重新停在方案阶段，但先快速复核当前环境事实是否仍匹配。
   - 可以直接改的条件：用户明确要求直接改，且改动范围、验收方式和回退边界已经清楚。
   - 如果已审核方案与当前事实冲突，以当前事实为准收窄实现，并在交付中说明偏差。例如线上元数据字段标识不同于方案草稿时，代码必须对齐当前元数据。
5. 实施改动：优先复用现有结构、服务和仓库现成能力；只有现有 helper、wrapper、标准库或成熟三方能力都不能满足时，才新增抽象、方法或依赖，并在交付中说明缺口。快速判断现有能力是否适用：先看函数签名/接口定义是否匹配，再看实现是否覆盖边界条件，最后看是否有已知坑（注释/测试/issue）。
6. 闭环检查：新增配置项时检查 schema、解析逻辑、默认值、文档；新增 API 时检查路由注册、权限验证、日志、错误码；新增参数/字段时逐一核对生产端、解析端、消费端以及关键回调/保存/展示链路是否都已接上。
7. 验证闭环：按可用性优先执行目标测试、相关测试、构建、脚本或样例；无法执行时说明原因、替代验证和未覆盖行为。
8. 交付结果：说明方案依据、改动文件、行为影响、验证结果、风险和后续建议。

## 门禁与降级
- 需求缺少关键输入输出、验收口径、权限/兼容边界或数据模型时，先列关键假设；高风险假设必须向用户确认。
- 需要改公共 API、数据结构、依赖、部署配置、权限模型或跨模块契约时，先说明影响和回滚/验证方式。
- 能通过配置、现有服务或既有扩展点完成时，不新增抽象或依赖。
- 仓库已有标准 helper、wrapper、脚手架、SDK 封装或成熟三方库能满足需求时，不再新写第二套方法或重复封装；确实不能复用时说明证据和最小新增范围。
- 需要新增共享 helper、wrapper、模板层或三方依赖时，先说明为什么现有能力和扩展点都不适用；会影响公共层或调用方时必须先让用户确认。
- 实现过程中发现本质是 bug 修复时，切换到 `fix-bug` 的根因闭环，不把回归修复包装成新功能。
- 如果改动跨入口、配置解析、目标页面/服务、回调链路等多段流程，交付前明确本轮已覆盖路径和范围外路径，避免默认视为全量收口。

## 代码注释策略
- 生成或修改任何编程语言的代码时，默认要为可维护单元补功能性注释或语言原生 docstring，让读者能快速理解职责、输入输出、边界、副作用和设计原因。
- 新文件、模块、类、公共函数/方法、公共接口、复杂私有函数、工具函数、关键算法、业务规则、事务/并发/异步/资源管理、异常边界和外部系统交互必须有稳定注释。
- 注释写“为什么、边界是什么、调用方需要知道什么”；不机械复述“循环列表、判断为空、返回结果”等代码表面行为。
- 简单局部语句不逐行注释；如果没有任何注释就会让新增代码一眼看不出头绪，必须补模块/函数/关键分支级注释。

## Guardrails
- 不跳过方案确认，除非用户明确要求直接改。
- 不做无关重构，不改变外部 API 或依赖结构，除非需求明确要求。
- 不把任务过程、沟通记录或交付口径写入代码注释、skills、操作说明或示例说明。
- 涉及代码、注释、文档或提交时，署名必须遵守全局规则：不用 AI，统一用 `anfeng`。
- 不重复实现仓库已有 helper、wrapper、标准服务或成熟三方能力；要新写工具方法时必须先确认现有能力无法覆盖。
- 不用“顺手优化”扩大行为面；性能、结构或样式调整必须服务于当前需求。

## Output
使用简体中文，先给结论：方案 → 实现 → 影响 → 验证 → 风险。
- 方案：目标、落点、输入输出、验收方式和关键取舍。
- 实现：改动文件、核心行为、兼容边界。
- 影响：接口、数据、配置、权限、性能或用户行为变化。
- 验证：执行命令、结果、覆盖范围；未执行时写替代验证。
- 风险：未确认假设、未覆盖路径和建议下一步。
方案阶段不声称已实现；实现后必须区分已验证和未验证内容。
