# Rubric 详细评分标准

## 结构维度详细说明

### 维度1：Frontmatter质量（8分）
- name规范：小写字母、数字、连字符，符合 Anthropic SKILL 规范
- description包含：做什么 + 何时用 + 触发词（中英文）
- 长度：≤1024字符
- 评分：每项缺失扣2-3分

### 维度2：工作流清晰度（15分）
- 步骤明确可执行：每步有动词、对象、条件
- 有序号：Phase/Step/数字编号
- 输入输出明确：每步说明需要什么、产出什么
- 评分：模糊步骤每处扣2分，缺少序号扣3分

### 维度3：边界条件覆盖（10分）
- 异常情况处理：文件不存在、网络失败、权限不足
- fallback路径：Plan A失败时的Plan B
- 错误恢复：如何回滚、如何重试
- 评分：每缺一类边界条件扣3分

### 维度4：检查点设计（7分）
- 关键决策前有用户确认：删除文件、发布、付费操作
- 防止自主失控：长时间运行、递归操作、批量修改
- 评分：高风险操作无检查点每处扣2分

### 维度5：指令具体性（15分）
- 不模糊：避免"处理"、"优化"、"改进"等空泛词
- 有具体参数/格式/示例：命令行参数、JSON格式、代码示例
- 可直接执行：不需要猜测或补充信息
- 评分：模糊指令每处扣2分，缺少示例扣3分

### 维度6：资源整合度（5分）
- references/scripts/assets引用正确
- 路径可达：相对路径正确、文件存在
- 评分：引用错误每处扣1分

## 效果维度详细说明

### 维度7：整体架构（15分）
- 结构层次清晰：章节划分合理、逻辑递进
- 不冗余不遗漏：无重复内容、覆盖完整
- 与花叔生态一致：符合团队风格和约定
- 评分：结构混乱扣5分，冗余/遗漏各扣3分

### 维度8：实测表现（25分）
用测试prompt跑一遍，按以下标准打分：

#### 评分方式
1. 为每个skill设计2-3个典型用户prompt
2. 用子agent执行 with_skill 与 no_skill baseline
3. 可用多个模型时至少覆盖：当前默认模型、更小/快模型、偏代码/推理模型
4. 对比输出质量，从以下角度打分：
   - 输出是否完成了用户意图？
   - 相比不带skill的baseline，质量提升明显吗？
   - 有没有skill引入的负面影响（过度冗余、跑偏、格式奇怪）？
   - 多模型结论是否一致？是否只有某个弱模型依赖该规则？

#### skill_delta 评分标准
- +2: 无skill失败或明显跑偏，有skill完成
- +1: 有skill质量更稳、更少遗漏
- 0: 两者基本持平
- -1: 有skill更冗余或更慢但仍可用
- -2: 有skill导致错误、逃避或过度停问

#### 维度8最终分数
按平均 skill_delta 和任务完成质量综合打分。不要只因输出"看起来规范"就给高分。

#### 干跑验证（fallback）
如果无法跑子agent（超时、环境限制），用干跑验证打分，标注 `dry_run`。干跑也必须记录 prompt_id、expected、with_skill_expected_behavior、baseline_expected_behavior、skill_delta、score_reason。

## Skill 有用性的判定

### 高价值规则
no_skill 经常遗漏、误判或走错工具；with_skill 明显改善，并且不显著增加冗余。

### 低价值规则
多个模型 no_skill 已稳定完成，with_skill 只是重复常识、系统规则或通用工程习惯；应压缩到一句，或移出 SKILL.md。

### 负价值规则
with_skill 更容易停问、逃避、过度约束、输出过程流水或忽略用户目标；优先删除或改成"主动补证后推进"的规则。

### 模型专属规则
只帮助小模型、不影响强模型的内容可以保留，但要写成短规则或引用 references，避免长期占用主上下文。
