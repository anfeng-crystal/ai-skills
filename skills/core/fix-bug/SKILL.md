# Fix Bug

> **Cross-platform Agent Skill** — Claude Code · OpenAI Codex · OpenCode · OpenClaw 通用。
> 跨平台 SKILL.md，遵循开放 Agent Skill 规范。

## 触发边界
- 用户提供报错、日志、异常现象、失败测试、回归、线上行为不符合预期，且目标是定位根因并修复时使用。
- 只解释代码或执行流程时改用 `explain-code`；只审查风险、找问题、给建议但不改代码时改用 `review-code`；新增能力、扩展接口或改变预期行为时改用 `implement-feature`。
- 混合请求按用户目标判定：说”修掉、解决、定位根因并改”时进入本 skill；说”只看看、review、找风险”时不要改代码。

## 快速工作流
1. 建立事实：确认现象、复现条件、报错栈、输入数据、影响范围、最近变更和当前期望。
2. 还原路径：优先复跑失败用例或最小复现；无法复现时用日志、调用链、数据快照和差异对比补证据。
3. 定位根因：沿入口、调用链、数据流、状态写入点、异常处理、事务、缓存、并发和边界值验证假设。
4. 设计修复：根因必须能解释全部已知现象；先检查问题附近是否已有标准 helper、wrapper、SDK 封装、共享工具或平台能力可以直接纠正行为；选择最小修改点，保持接口、数据结构和调用方行为稳定；自检修复是否会引入新的边界条件（例如：补空值检查后，是否需要补默认值处理）。
5. 实施修复：只改问题附近必要逻辑；仓库已有规范 helper 或标准能力时优先回归到现有实现，不顺手新增第二套 util；需要兼容旧数据、旧配置或旧调用方时，把兼容边界写进测试或稳定注释。
6. 验证闭环：改动影响面小（单文件、无公共 API 变更）时只跑相关测试或失败用例；改动影响公共层（共享 helper、wrapper、公共 API）时必须跑全量测试或相关模块测试；没有测试时跑覆盖改动路径的构建、编译、脚本或人工验证步骤。
7. 交付结论：说明根因、修改、已验证内容、未验证风险和建议的下一步。

## 门禁与降级
- 根因未确认前不改代码；只能提出需要补充的日志、数据、复现步骤或候选假设。
- 修复需要改依赖、公共 API、文件结构、数据模型、外部系统协议或大范围重构时，先说明必要性、影响和验证方式，并等待用户确认。
- 复现失败但证据足够时，可以做最小防御性修复；证据不足时停止在排查建议，不用猜测性 patch 代替定位。
- 仓库已有规范 helper、wrapper、SDK 封装或标准能力能覆盖修复时，不再新写临时工具函数或第二套处理逻辑；现成能力不适用时说明原因。
- 修复如果需要新增共享 helper、retry/http/date 工具、wrapper 或公共扩展点，先说明为什么现有共享能力无法覆盖；会影响公共层时必须先让用户确认。
- 验证命令无法执行时，说明失败原因、替代验证、未覆盖行为和手工联调步骤。

## 代码注释策略
- 修改任何编程语言的代码时，必须检查新增或变更处是否需要功能性注释，让读者能快速看懂职责、边界、副作用和长期原因。
- 新增或改动文件、类、公共函数/方法、复杂私有函数、关键算法、事务/并发/异步/资源管理、兼容边界、异常边界和平台约束时，必须补稳定注释或语言原生 docstring。
- 修复 bug 时，注释只写兼容边界、失败条件、业务口径或不直观原因；不写“本次修复、经过排查、临时处理、为了解决问题”这类过程记录。
- 简单赋值、简单透传、显而易见的条件判断不逐行注释；如果代码整体不易读，优先用命名、拆分和类型表达，再用注释补充意图。

## Guardrails
- 不做无关重构，不改依赖、外部 API 或文件结构，除非修复必须且用户确认。
- 不吞异常，不返回无说明空数据，不用“先兜住再说”的改动掩盖根因。
- 不把排查过程、尝试路径或修复经过写入代码注释、skills、操作说明或示例说明。
- 涉及代码、注释、文档或提交时，署名必须遵守全局规则：不用 AI，统一用 `anfeng`。
- 不为了补 bug 顺手新造 helper、wrapper 或重复封装现有标准能力；确需新增时必须说明为什么现有方案无法覆盖。
- 不把 review 结论当作已修复结果；没有实际修改和验证时必须明确说明。

## Output
使用简体中文，先给结论：原因 → 修改 → 验证 → 风险。
- 原因：根因、证据来源、为什么能解释现象。
- 修改：改动文件、最小行为变化、兼容边界。
- 验证：执行命令、结果、覆盖路径；未执行时写替代验证。
- 风险：未验证内容、依赖外部条件、建议下一步。
