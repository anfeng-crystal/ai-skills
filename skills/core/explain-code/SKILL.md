# Explain Code

> **Cross-platform Agent Skill** — Claude Code · OpenAI Codex · OpenCode · OpenClaw 通用。
> 跨平台 SKILL.md，遵循开放 Agent Skill 规范。

## 触发边界
- 用户要求解释代码、梳理流程、说明设计意图、分析函数或模块职责、理解错误原因但未要求修复时使用。
- 不修改代码；需要定位并修复现有错误时改用 `fix-bug`，需要审查问题或风险时改用 `review-code`，需要新增或改造能力时改用 `implement-feature`。
- 混合请求按动作判定：问“为什么、怎么走、这段做什么”时解释；问“哪里有问题、有什么风险”时审查；问“修掉、实现、改成”时切到对应 skill。

## 快速工作流
1. 确认解释对象和模式：用户问"这个文件/模块做什么"用模块解释模式；问"这个函数/方法怎么用"用函数解释模式；问"为什么这样写/设计意图"用设计意图解释模式；问"这段代码怎么走"用执行流程解释模式；确认是否只读（不修改代码）。
2. 建立证据（按优先级）：入口优先（主函数、路由、事件处理、公共 API）> 调用关系（调用方 > 被调用方）> 失败路径优先（如果是 bug 排查背景：异常处理、错误日志、边界条件）> 数据流（关键数据结构、配置、外部依赖）> 测试和文档 > 共享能力（仓库已有 helper、wrapper、SDK 封装、模板或标准能力）；如果相关文件过多（>5 个），先读入口和调用方，再根据用户关注点补充；缺材料时列出假设和需要补充的文件。
3. 按模式组织说明：职责 → 流程 → 设计原因 → 边界；复杂逻辑补充事务、缓存、并发、平台约束或算法原因。
4. 收口：列出易错点、可验证入口、未确认假设和建议继续阅读的位置。

## 门禁与降级
- 只读解释时不改代码、不生成补丁、不把建议当作已完成修改。
- 没有可读代码、文件路径、片段或调用入口时，不做具体实现判断；先给材料清单或只解释可确认的通用概念。
- 上下文不足会影响结论时，判断假设影响范围：关键分支判断、数据流转、异常处理依赖假设时必须标注假设，说明”如果假设不成立，结论会如何变化”；局部实现细节、命名推断、常规模式可以保守推断，但要说明”基于常规实现推断”。
- 关键证据缺失且会误导用户时，先请求或列出需要补充的文件、日志或运行信息。
- 发现明显 bug 或风险时，只能作为”可能问题”提示；用户要求修复或审查时切换到 `fix-bug` 或 `review-code`。

## Guardrails
- 不做无关扩展，不把猜测写成事实。
- 不用“应该、大概、可能就是”包装成事实；每个关键判断要能对应到代码、配置、测试或用户输入。
- 不输出与问题无关的大段背景；用户只问局部时保持局部。
- 发现仓库已有标准 helper、wrapper、SDK 封装或模板与当前实现强相关时，应点明复用关系；如果进一步判断这是重复造轮子，只作为设计提示说明，需要正式定性时切到 `review-code`。
- 如果用户的问题本质是在问“为什么没有自己再造一套”，应先说明共享层职责、调用入口和复用收益，再展开局部实现细节。

## Output
使用简体中文，按需压缩：概述 → 入口/流程 → 设计原因 → 关键边界 → 假设/需补充。
- 概述：一句话说明对象职责和调用时机。
- 入口/流程：列入口、关键调用、数据/状态变化和返回。
- 设计原因：只解释能从代码、配置、测试或用户输入支撑的原因。
- 关键边界：异常、并发、事务、缓存、权限或平台限制。
- 假设/需补充：列缺失文件、运行信息或仍无法确认的判断。
