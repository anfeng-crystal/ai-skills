# Review Code

> **Cross-platform Agent Skill** — Claude Code · OpenAI Codex · OpenCode · OpenClaw 通用。
> 跨平台 SKILL.md，遵循开放 Agent Skill 规范。

## 触发边界
- 用户要求 review、审查、质量检查、风险分析、找问题、看 PR 或给修改建议时使用。
- 默认不修改代码；用户要求落实修复现有错误时切换到 `fix-bug`，要求新增或改造能力时切换到 `implement-feature`，只问代码含义时切换到 `explain-code`。
- 混合请求按动作判定：说“找风险、有没有问题、review 一下”时只审查；说“直接修”时不要停留在 review。

## 快速工作流
1. 确认审查范围：变更文件、diff、入口、调用链、依赖边界、测试和用户特别关注点。
2. 建立上下文：读取相关代码、调用方、被调用方、配置、数据模型和现有测试；缺少 diff 或代码时先说明可审范围。
3. 选择审查维度（按优先级）：所有变更必查正确性（边界条件/空值/状态转换/错误处理）和兼容性（接口/数据结构/配置/调用方）；公共层变更必查可维护性（职责边界/命名/重复逻辑/注释质量）、工具复用（是否重复造轮子）、安全/权限（认证/授权/数据访问）；性能敏感路径查性能（循环/查询/分页/缓存/资源释放）、并发/事务（锁/竞态/隔离级别）。
4. 输出 findings（按严重程度排序）：严重问题（会导致功能错误、数据错误、安全/权限问题、构建失败或线上事故）、一般问题（影响兼容性、性能、可维护性或局部稳定性）、建议优化（不阻塞交付，但能降低维护成本）；每条必须包含：位置（文件:行号或函数名）→ 影响（会导致什么）→ 依据（从代码/配置/测试看到的事实）→ 建议（可执行的修改方向）。
5. 无问题分支：没有发现问题时明确说明，并列出已看范围、未覆盖范围和仍建议补的测试。
6. 收口：列出无法验证处、开放问题和最小后续动作；不要把建议写成已完成修复。

## 门禁与降级
- 没有具体代码、diff 或文件范围时，不做泛泛 review；先请求范围，或在标题中标明”有限审查”并只给可由现有信息支持的风险。
- 每条 finding 必须能追溯到文件、函数、类、配置或明确代码范围；无法定位的问题只能列为开放问题或测试缺口。
- 发现需要立即修复的问题时仍保持只读，除非用户明确要求进入修复或实现。
- 如果审查结果依赖运行时、数据、权限或外部系统，必须标注未验证假设。
- 如果变更新增共享 helper、wrapper、三方依赖或第二套基础设施实现，判断理由：理由仅是”方便当前需求”且现有公共层能覆盖时按高风险可维护性问题审查（重复造轮子）；现有公共层确有缺口且补充理由充分时视为可能合理，但仍需审查接口设计和命名。

## References

- 问题输出格式：`references/finding-format.md`，用于 finding 的位置、影响、依据和建议。

## Guardrails
- Findings 优先，摘要靠后。
- 每个问题必须有具体位置或范围，并说明影响。
- 不写空泛评价，不把过程叙述当作审查结论。
- 不直接改代码，不把“建议修改”表述成“已经修改”。
- 不用泛泛表扬填充输出；没有发现问题时重点说明覆盖范围和剩余风险。
- 公共接口、复杂逻辑、跨模块工具、业务规则、并发/事务/资源/异常边界缺少稳定注释时，应作为可维护性问题提出；机械复述、过程记录和变更流水也要指出。
- 仓库、标准库、官方 SDK、共享 helper、wrapper、模板或成熟三方能力已经能覆盖需求，但变更又新写了一套方法、工具类或包装层时，应作为“重复造轮子”问题指出。
- 代码、注释、文档或提交信息署名 AI，或需署名时未使用 `anfeng`，应作为问题指出。

## Output
使用简体中文：严重问题 → 一般问题 → 建议优化 → 测试缺口。
- 每条 finding 使用：位置 → 影响 → 依据 → 建议。
- 严重问题：会导致功能错误、数据错误、安全/权限问题、构建失败或线上事故。
- 一般问题：影响兼容性、性能、可维护性或局部稳定性。
- 建议优化：不阻塞交付，但能降低维护成本。
- 没有问题时输出“未发现阻塞问题”，并写明已看范围、未覆盖范围和剩余风险。
