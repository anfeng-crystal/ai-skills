# 苍穹问题原因分析参考

## 适用场景
- 生产或研发环境故障排查。
- 性能瓶颈、内存、线程、数据库连接池、缓存、部署发布问题。
- 代码缺陷、配置差异、依赖版本变化导致的异常。

## 排查流程
1. 定义问题：现象、影响范围、发生时间、触发条件、最近变更。
2. 收集证据：应用日志、平台日志、监控指标、告警、git 变更、配置差异。
3. 建立时间线：把日志、监控、发布、操作记录按时间对齐。
4. 提出假设：代码缺陷、配置问题、资源耗尽、依赖故障、数据问题。
5. 验证假设：复现、查日志证据、对比正常环境、最小化验证。
6. 输出结论：直接原因、根本原因、止血方案、根治方案、验证步骤、预防措施。

## 常用证据方向
- 日志关键字：`ERROR`、`Exception`、`Timeout`、`OOMKilled`、`Connection pool`、`deadlock`。
- JVM：线程 dump、GC 日志、堆 dump、CPU/内存趋势。
- 数据库：慢查询、连接池、索引、锁等待、事务边界。
- 苍穹二开：插件事件时机、跨库事务、DataSet 生命周期、字段标识、元数据配置。

## 列表或页面无数据
- 先还原用户实际入口：URL 参数、`formId`、`billFormId`、移动/PC 页面类型、登录用户、角色和组织。
- 优先追踪过滤链路，不要把并发出现的远程调用超时、Dubbo 本地调试超时或后台异步错误直接当成无数据根因。
- 按顺序核对过滤来源：页面/列表元数据默认过滤、用户常用/方案过滤、URL 参数插件、列表插件 `setFilter`、平台查看权限、组织维度权限、数据权限和操作规则。
- 涉及权限时，必须区分“角色有菜单/操作权限”和“最终查询 QFilter 有数据”；查看权限、组织维度、数据权限和操作规则会被平台组合后注入 ORM 查询。
- 需要给出 SQL 证据时，围绕同一个 traceId 展示过滤配置查询、权限/操作规则查询和最终业务表 SQL；若日志因长 `IN` 条件截断，明确说明可见部分和不可见尾部。
- 若 SQL 出现 `1!=1`、空 `IN` 或恒假条件，先查上游 QFilter 来源和空集合生成点；不要只从结果 SQL 反推某个业务插件生效。
