# 关键事实停问与离线兜底

这份规则只管一件事：当任务继续推进需要确认某个事实时，先判断它是不是硬依赖，再决定是本地继续、在线增强，还是停下来问用户。

## 任务分解

先把任务拆成这些最小项，只保留当前实现或结论真正会用到的部分：

- 目标对象：插件类型、单据/列表/报表/工作流、事件点、调用链。
- 关键标识：`entityId`、`formId`、`refType`、字段标识、配置键。
- 平台能力：API 类名、方法签名、参数、返回结构、事务边界。
- 数据口径：主表/子表、组织/租户、权限范围、保存目标。
- 验证方式：编译、单测、脚本查询、post-check、人工确认。

## 关键事实清单

以下事实只要不明确，就要优先确认：

- `entityId`
- `formId`
- `refType`
- 字段标识与字段类型
- API 类名、方法重载、事件方法名
- 事件时机与生命周期位置
- 配置项是否存在、取值范围、默认值
- 组织、租户、权限、数据落点口径
- 返回结构、主键、枚举映射、状态值

## 硬依赖判定

满足任一条，就算硬依赖：

- 会改变代码分支，例如保存前/保存后、列表插件/操作插件、`save`/`submit`。
- 会改变目标对象，例如 `entityId`、`formId`、单据类型、基础资料实体。
- 会改变平台 API 选择，例如该用哪个 helper、service、query、filter 或上下文对象。
- 会改变数据落点，例如主表/子表、当前单据/来源单据、组织内/跨组织。
- 会改变事务、权限或租户口径。
- 会影响不可逆或高风险结果，例如删除、批量更新、提交流程、审批动作、回写。

只要缺失事实会让实现只能“选错”，而不是还能保守推进，就必须停问。

## 可延后与可保守假设

可延后，不必停问：

- 命名、注释、日志文案、异常提示文案。
- 方法是否再抽一层，只要不改行为。
- post-check 的附加项。
- 代码组织方式，只要不改语义。

可以保守假设，但要满足三点：

- 有本地证据支撑。
- 假设可回滚，不会牵连大量代码。
- 交付时能明确标记为“假设”，不是确认事实。

## 本地证据等级 A-E

从强到弱建议这样看证据：

### A 级：仓库内显式事实

来源：

- 当前代码
- `SKILL.md`
- `references/`
- `scripts/`
- 项目配置
- 常量
- 模板

适用：

- 插件类型
- 事件点
- 字段名
- 方法名
- 配置项存在性
- 调用顺序

### B 级：同仓库同类实现

来源：

- 同模块
- 同类型
- 最近实现
- 相似插件、相似页面、相似单据

适用：

- API 用法
- 事件时机
- 字段处理方式
- 异常处理模式
- 返回值约定

### C 级：脚本本地结果

来源：

- `cosmic-api-knowledge.py`
- `cosmic-form-metadata.py`
- `cosmic-basedata-query.py`
- 本地缓存或离线返回

适用：

- 类或方法存在性
- 元数据缓存
- 基础资料映射
- 历史知识点

说明：

- 只把它当作离线知识能力，不当作在线实时真相。
- 只通过脚本使用离线知识能力或内置缓存，不鼓励把 SQLite 直接当业务查询入口。

### D 级：构建产物与依赖痕迹

来源：

- `gradle compile/test`
- jar
- 生成文件
- 依赖声明
- 编译错误信息

适用：

- 类是否可编译
- 签名是否匹配
- 引用是否存在

### E 级：用户输入

来源：

- 用户明确给出的 `entityId`
- 用户明确给出的 `formId`
- 用户明确给出的 `refType`
- 用户明确给出的字段口径

适用：

- 任务范围
- 业务意图
- 明确标识

## 在线查询策略

- 在线元数据/基础资料查询是可用时增强，不是必需前置。
- VPN、代理或网络限制导致在线查询不可用时，不视为失败。
- 在线不可用时，继续使用 A/B/C/D 级证据推进，不要编造 `entityId`、`formId`、`refType`、字段、API 签名或返回结构。
- 如果当前任务必须依赖的关键事实本地确实无法确认，才停下来问用户。

## 证据冲突处理

- A 级与其他等级冲突时，以 A 级为准；如果冲突点正好影响当前任务正确性，就停问。
- 同等级来源对关键事实给出不同结论，且无法通过源码、模板、历史实现解释时，停问。
- 只有 C/D 级证据，且事实决定业务口径、对象或事件时，不直接定稿。
- 事实只影响增强信息、展示细节或后续优化时，可以先继续，并标记风险。

## 决策矩阵

| 状态 | 处理方式 |
|---|---|
| 已确认且不冲突 | 继续推进 |
| 未确认，但不影响当前步骤 | 先按占位推进，标记待确认 |
| 未确认，且影响当前步骤 | 先查本地证据 |
| 本地可确认 | 继续推进 |
| 本地不可确认，但只影响在线增强 | 降级继续 |
| 本地不可确认，且继续会选错对象、事件、接口或数据落点 | 停下来问用户 |

## 问用户模板

只问一个最小问题，直接指向缺失事实：

- `当前缺少 <事实名>，这会影响 <具体步骤或判断>。请确认：<给出 1-2 个可选值或直接提供值>。`
- `这里的 <事实名> 不能从本地确认。为了避免选错 <对象/事件/API>，请直接给出准确值。`
- `如果按当前实现继续，需要先确认 <事实名>。你给我一个确定值，我就按这个走。`

## 兜底原则

- 先看仓库源码和同类实现。
- 再看 `references/`、`scripts/` 和本地缓存结果。
- 在线能力可用时才做增强确认。
- 在线不可用时，继续基于已确认事实推进。
- 只有当缺失事实会直接影响当前任务正确性，且本地证据无法补齐时，才停问。
