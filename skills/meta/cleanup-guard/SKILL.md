# Cleanup Guard

> **Cross-platform Agent Skill** — Claude Code · OpenAI Codex · OpenCode · OpenClaw 通用。
> 用于收尾阶段的安全清理，不负责功能实现、代码修复或提交发布。

## 触发边界
- 用户要求收尾、清理测试残留、删除未跟踪文件、压缩工作区脏区时使用。
- 重点处理 `git status` 里的未跟踪文件、临时测试文件、`.gitkeep`、调试脚本、临时日志和本次任务生成的目录。
- 不负责决定业务文件是否保留；归因不清时必须停住确认。

## 本轮清理行为统计
每次进入收尾清理，先判断这轮属于哪一类：

1. `delete_generated_temp`
   - 明确删除本次任务生成的临时产物
   - 目标是安全回收测试残留和联调垃圾

2. `compress_workspace_noise`
   - 不一定删除文件
   - 重点是把工作区脏区分类清楚，降低后续提交和排查噪音

3. `no_cleanup`
   - 本轮没有足够证据证明某些文件可删
   - 只做分类和建议，不执行删除

默认优先级：
`compress_workspace_noise` > `delete_generated_temp` > `no_cleanup`

也就是说，先把工作区讲清楚，再决定删不删；不是先删再整理。

## 核心规则
- 未跟踪文件默认视为用户资产，不因名字像测试文件就直接删除。
- 只有满足以下条件之一，才能进入“可安全删除”候选：
  - 已确认由本次任务生成
  - 明确是临时测试产物，且不在提交范围内
  - 用户点名要求删除
- 删除前必须先给出“可安全删除清单”，经用户确认后再执行。
- 不能先删后核对；不能把“测试命名”当成删除证据。

## 候选判定顺序
对每个删除候选，按下面顺序判断：

1. 是否为已跟踪文件
   - 是：默认不删

2. 是否为未跟踪源码文件
   - 是：默认归为 `needs_confirmation`

3. 是否有本次任务生成证据
   - 有：可进入 `generated_temp`

4. 是否只是名字像测试文件
   - 是：证据不足，不能直接删

5. 是否被用户点名删除
   - 是：进入待执行清单，但仍要回显删除范围

默认要求每个候选文件只归入一个主分类，避免同一文件同时被说成“可删”又“待确认”。

## 快速工作流
1. 先收集工作区现状：

```bash
git status --short
```

2. 把候选文件分成四类：
   - `tracked_changes`：已跟踪修改，默认不删
   - `untracked_user_owned`：归因不明或明显是用户文件，默认不删
   - `generated_temp`：本次任务生成的临时文件，可进入删除候选
   - `needs_confirmation`：看起来像测试残留，但归因未确认，必须停住

3. 对每个候选文件补最小证据：
   - 文件路径
   - 文件类型或目录类型
   - 归因依据（本次创建、测试输出、用户已有、无法确认）
   - 是否已进入提交范围

4. 输出清单，只列两部分：
   - `建议删除`
   - `保留/待确认`
   需要稳定格式时，直接套用 `references/safe-delete-report.md` 的输出模板和判定口径。

5. 只有用户明确确认后，才执行删除。

6. 删除后再次核对：

```bash
git status --short
```

并报告实际删除结果和剩余脏区。

## 归因检查
- 对未跟踪文件，优先看内容、所在目录、引用关系、生成命令、当前任务上下文。
- `git log` 查不到未跟踪文件历史，不能作为“可删除”的正向证据。
- 如果文件已被创建在正式源码目录下，更要保守处理，先看内容再判断。
- `.gitkeep` 只有在确认目录是本次临时创建且不再需要时才删。
- 测试目录、测试脚本、临时插件、联调样例，只有在确认未进入提交范围且为本次生成时才删。

## 通用化原则
优化这类清理规则时，默认写成“证据判断 + 删除动作”的通用准则，不写成一次性案例。

反例：
- “删除 TestPlugin.java 前先确认”

正例：
- “删除未跟踪源码文件前，必须先完成归因确认；不能因为名字像测试文件就直接删除”

如果一条规则只能解释某一次清理动作，就不该直接进入长期 skill。

## 保留门槛
一次清理建议只有同时满足以下条件，才进入“建议删除”：

1. 文件不在提交范围内
2. 有本次任务生成或临时测试产物的证据
3. 归因不依赖单一命名猜测
4. 用户已确认删除范围

如果只能证明“看起来像能删”，不能证明“删了是安全的”，默认保留。

## 用户确认检查点

| 场景 | 停住确认内容 |
|------|-------------|
| 发现未跟踪源码文件 | "该文件是未跟踪源码文件，归因未确认，默认不删除。是否保留？" |
| 发现名字像测试文件但来源不明 | "名称像测试文件，但不能证明是本次生成。是否继续保留？" |
| 删除范围包含目录 | "将删除目录及其内容，确认范围无误后再执行。" |
| 计划删除 3 个以上候选 | "以下是可安全删除清单，请确认后执行。" |
| 用户要求‘全删’但存在归因不明文件 | "存在归因不明文件，建议先排除这些文件后再删。" |
| 候选位于正式源码目录 | "该文件位于正式源码目录，默认按用户资产处理。是否继续保留？" |

## Guardrails
- 不删除任何已跟踪文件，除非用户明确要求。
- 不删除归因不明的未跟踪源码文件。
- 不把 `Test*`、`tmp*`、`.gitkeep` 这类命名模式当作充分证据。
- 不把“未进入提交”自动等同于“可删除”。
- 不把一次性案例文件名直接写成长期删除规则。
- 删除动作完成后必须回显剩余工作区状态。

## Output
使用简体中文，先给结论：本次建议删除范围 → 保留/待确认范围 → 删除依据 → 执行后结果。

需要统一口径时，优先复用：
- 输出模板：`references/safe-delete-report.md`

推荐输出格式：
```
**本轮清理行为统计**
- delete_generated_temp: ...
- compress_workspace_noise: ...
- no_cleanup: ...

**建议删除**
- path: ...
  reason: 本次联调生成的临时测试文件，未进入提交范围

**保留/待确认**
- path: ...
  reason: 未跟踪源码文件，无法确认是否为用户资产

**下一步**
- 如确认删除，我将只删除“建议删除”部分
```
