# 苍穹插件开发流程

参考 `references/base/plugin/*`，AI 在编写任何苍穹插件时，默认按以下顺序推进：

## 1. 识别应用场景，先判断插件类型

- 先判断需求属于哪一类：表单/单据 UI、列表、左树右表、操作服务、BOTP 转换、反写、报表、打印、OpenAPI、后台任务、工作流、导入。
- 优先选择已有封装基类；只有封装层未覆盖时，才回退到原生插件基类。
- 若需求本质是"界面联动/控件交互"，优先考虑表单、单据、列表类插件；若需求本质是"事务内校验/落库/状态流转"，优先考虑操作插件。

## 2. 定位合适的事件方法，不要先写实现

- 先在对应的 `references/base/plugin/plugin-*.md` 中确认事件清单、触发时机和适用边界，再决定重写哪个事件。
- 有交互界面的场景，要区分"表单事件"和"控件事件"；控件事件通常还需要 `implements XxxListener` 并在 `registerListener` 中注册。
- 无交互界面的场景（如操作、转换、反写、任务），重点是按生命周期选择正确事件，不要套用 UI 插件的写法。

## 3. 先读模板，再定代码骨架

- 确认插件类型后，必须先读取对应 `assets/*.java` 模板，沿用已有方法签名、命名风格和注释结构。
- 模板里没有的事件，不要凭印象补写；先查参考文档，再用脚本验证签名。

## 4. 按需求拆解"要用什么上下文"

- UI 控制：优先通过 `this.getView()` 和控件编程模型完成，如启用/禁用、隐藏、弹窗、刷新、打开页面。
- 界面数据：优先通过 `this.getModel()` 读写字段、分录和数据包。
- 操作事务：优先使用操作插件上下文、`OpUtils`、`OperateChain` 等封装处理状态流转和事务内逻辑。
- 查询、元数据、DynamicObject、附件等能力，先走 `references/adv/*` 封装文档，再决定是否需要原生 SDK。

## 5. 补齐事实来源，再写实现

- 字段标识、表单元数据不确定时，先调用 `cosmic-form-metadata.py`。
- `@Override` 方法、BOS SDK 关键方法、原生类名不确定时，先调用 `cosmic-api-knowledge.py` 校验。
- 只有在"插件类型 + 事件方法 + 字段/SDK 签名"都确认后，才开始生成代码。

## 6. 实现时遵守最小干预原则

- 插件只做当前事件应该做的事，不要把重查询、复杂编排、跨模块业务堆进 UI 事件。
- 能用封装就不要退回原生；能复用现有模板和工具方法，就不要重新发明样板。
- 对控件事件，记得同时处理"implements 监听接口"和"registerListener 注册"这两个动作。

## 7. 生成后自检

- 反查一遍：插件类型是否选对、事件是否选对、是否误用了不存在的事件、是否遗漏监听注册、是否把 UI 逻辑写进事务插件、是否把事务逻辑写进 UI 插件。
- 检查 import 是否最小化、字段是否准备齐全、是否遵守"封装优先，原生兜底"。

## 8. 自动校验（强制）

- 对生成的 `.java` 文件执行 `$KINGDEE_COSMIC_SKILL_ROOT/scripts/cosmic-post-check.py --fix-hint`，自动检查编译错误、幻觉方法名、幻觉类名、场景错配、编码偏好等问题。
- 若存在 **ERROR**：立即根据 fix-hint 修复，修复后重新执行脚本确认通过。
- 若存在 **WARNING**：优先修复；确有合理理由可保留，须在代码注释中说明。
- 形成"**生成 → 校验 → 修复 → 复检**"闭环，不允许跳过此步骤。
